## 🌙 ภาพรวมโปรเจกต์ (Project Overview)

**Lunar Project - Student Attendance System (SAS)** คือระบบจัดการการเข้าเรียนและการเช็คชื่อของนักเรียน ออกแบบหน้าจอตามสไตล์ **Neubrutalist** (เน้นการใช้สีสันสดใสพาสเทล, ขอบเส้นหนาสีดำคมชัด, เงาแนวระนาบแบนทึบไม่ฟุ้ง และฟอนต์ที่อ่านง่ายเป็นมิตร) โดยระบบทั้งหมดถูกพัฒนาขึ้นในรูปแบบของ Single Page Application (SPA)

### เทคโนโลยีที่ใช้งาน (Tech Stack)
- **Frontend**: Vanilla HTML5, Vanilla JavaScript (ES6+), Vanilla CSS ร่วมกับ Tailwind CSS v4
- **Build Tool**: Vite (รองรับการบิลด์หน้าเว็บแบบ Static)
- **Backend & Database**: Supabase (จัดการการล็อกอินและสิทธิ์การเข้าใช้งาน Auth, PostgreSQL Database สำหรับเก็บข้อมูล และ Storage Buckets สำหรับเก็บไฟล์รูปภาพ)

## 🏗️ สถาปัตยกรรมระบบและการกำหนดเส้นทาง (Architecture & Routing)

โปรเจกต์นี้ทำงานแบบ **Single Page Application (SPA)** โดยควบคุมเส้นทาง (Routing) ด้วยระบบ Hash-based routing ผ่านไฟล์ `src/main.js`

1. **Route Mapping**: ค่าคงที่ `ROUTES` ในไฟล์ `src/main.js` จะเป็นตัวจับคู่ระหว่างค่า URL Hash (เช่น `#admin-dashboard`) กับ HTML template และฟังก์ชันในการเริ่มต้นหน้าจอนั้นๆ (init function)
2. **HTML Templates**: ถูกดึงเข้ามาใช้งานในรูปแบบของดิบ (raw string literals) โดยใช้ฟีเจอร์ของ Vite ผ่านคำสั่ง `?raw` (ตัวอย่างเช่น `import SharedSettingPage from './pages/shared/setting/setting.html?raw'`)
3. **Role Gating (การกั้นสิทธิ์ผู้ใช้งาน)**: การตรวจสอบสิทธิ์จะใช้ตัวแปร `allowedRoles` ที่ระบุใน `ROUTES` โดยตัวระบบ Router จะตรวจสอบบทบาทของผู้ใช้จากตาราง `profiles` ทุกครั้งที่มีการเปลี่ยนหน้าจอ และนำทางผู้ใช้ที่ไม่มีสิทธิ์กลับไปยังหน้าหลักของตนเองทันที
4. **Initialization Pattern**: แต่ละหน้าจะส่งออกฟังก์ชันเริ่มต้น (เช่น `initSetting(role, avatar)`) ซึ่งทำหน้าที่ตั้งค่า Event Listeners ผูกปุ่มคำสั่ง ดึงข้อมูลเฉพาะของหน้านั้นจากฐานข้อมูล และแสดงผลในรูปแบบ dynamic UI

## 📂 โครงสร้างโฟลเดอร์ (Directory Structure)

- **`src/auth/`**: โฟลเดอร์เกี่ยวกับการล็อกอินและลงทะเบียน
  - `#login` & `#register`: หน้าเข้าสู่ระบบและหน้าสมัครบัญชีของนักเรียน
  - `#admin-register`: หน้าลงทะเบียนสำหรับ Admin ใช้สร้างบัญชีให้ครูหรือผู้นำห้องเรียน
- **`src/lib/`**: โฟลเดอร์รวมสคริปต์ช่วยเหลือและตัวแปรส่วนกลาง
  - `supabaseClient.js`: ส่วนเชื่อมต่อกับ Supabase API โดยอ่านค่าคีย์จาก `.env`
  - `ui.js`: ฟังก์ชันช่วยเหลือของระบบส่วนกลาง เช่น การแสดงแจ้งเตือนบับเบิ้ล (`showToast()`) และตัวกรองป้องกันการโจมตีฝังโค้ด (`escapeHTML()`)
- **`src/pages/`**: แดชบอร์ดตามสิทธิ์ผู้ใช้ และโมดูลการใช้งานหน้าเว็บที่ใช้ร่วมกัน
  - **Role Dashboards**: หน้าแดชบอร์ดสรุปผลเฉพาะของแต่ละบทบาท (ได้แก่ `admin/dashboard`, `teacher/dashboard`, `leader/dashboard`, `student/dashboard`)
  - **`shared/`**: ยุบรวมและแชร์หน้าจอที่มีการทำงานคล้ายกันของแต่ละบทบาทให้อยู่ในโมดูลเดียวเพื่อความง่ายในการดูแลรักษา:
    - `activity`: หน้าแสดงผลและบันทึกประวัติกิจกรรม/รายวิชาเรียนย้อนหลัง
    - `approve`: หน้าหลักในการตรวจเช็ค อนุมัติ และปฏิเสธการเข้าเรียน (รองรับทั้งการอนุมัติรายบุคคล และปุ่มอนุมัติทั้งหมด "Verify All" ที่มาพร้อมระบบนับถอยหลังป้องกันความผิดพลาด)
    - `attendance`: หน้าจอบันทึกการเข้าชั้นเรียนและส่งใบลาของนักเรียน
    - `schedule`: หน้าแสดงตารางสอน (สำหรับครูและผู้นำจะเป็นแบบอ่านอย่างเดียว ส่วนแอดมินสามารถสร้างและแก้ไขได้)
    - `setting`: หน้าตั้งค่าและอัปเดตข้อมูลส่วนตัว (เชื่อมต่อกับ Supabase Storage ถัง `avatars` และตาราง `user_assets` พร้อมระบบครอบตัดภาพโปรไฟล์)
    - `user`: หน้าต่างจัดการข้อมูลประวัติผู้ใช้และข้อมูลสิทธิ์ในระบบ
  - **`error/`**: หน้าจอแจ้งข้อผิดพลาด 404 หรือหน้าแสดงข้อผิดพลาดการเชื่อมต่อเครือข่าย
  - **`parent_check/`**: หน้าจอตรวจสอบประวัติเข้าเรียนแบบด่วนสำหรับผู้ปกครอง (จำกัดสิทธิ์แบบอ่านอย่างเดียว)

## 🗄️ โครงสร้างตารางฐานข้อมูลหลัก (Database Schema Summary)

ฐานข้อมูลหลักถูกติดตั้งและรันอยู่บน Supabase มีรายละเอียดตารางหลักๆ ดังนี้:

- **`profiles`**: เก็บข้อมูลผู้ใช้งานระบบ เชื่อมต่อโดยตรงกับระบบ Supabase Auth (`id` อ้างอิงจาก `auth.users.id`)
  - คอลัมน์ที่สำคัญ: `id`, `firstname`, `lastname`, `role` (`student`, `teacher`, `leader`, `admin`), `stu_id` (รหัสนักเรียน), `class_id` (รหัสชั้นเรียน/ห้อง), `roll_number` (เลขที่ประจำตัวนักเรียน)
- **`attendance_logs`**: ตารางที่จัดเก็บประวัติการเช็คชื่อเข้าชั้นเรียนและประวัติการลาทั้งหมดแบบเบ็ดเสร็จ
  - คอลัมน์ที่สำคัญ: `id`, `student_id`, `subject_id`, `status` (`Present` = มาเรียน, `Sick` = ป่วย, `Personal` = ลากิจ, `Activity` = กิจกรรม), `notes` (หมายเหตุ/รายละเอียดเหตุผลการลา), `period` (`Full Day`, `Morning`, `Afternoon`), `verification_status` (`pending`, `approved`, `rejected`), `verifier_id` (ผู้ตรวจสอบ), `verified_at` (เวลาที่ตรวจ), `reject_reason` (เหตุผลหากโดนปฏิเสธ)
- **`schedule`**: ตารางแสดงข้อมูลรายละเอียดตารางการเรียนการสอน
  - คอลัมน์ที่สำคัญ: `id`, `subject` (วิชาเรียน), `room` (ห้องเรียน), `day` (วันเรียน), `start_time`, `end_time`, `class_id`
- **`user_assets`**: ลิงก์เก็บรูปโปรไฟล์โปรเจกต์ที่อัปโหลดเข้า Storage
  - คอลัมน์ที่สำคัญ: `id`, `user_id`, `asset_type` (เช่น `'avatar'`), `url`

## 💡 แนวทางการพัฒนาและข้อพิจารณา (Developer Guidelines & Patterns)

ในการแก้ไขหรือปรับปรุงโค้ดโปรเจกต์นี้ ขอให้ยึดข้อปฏิบัติเหล่านี้เสมอ:

1. **การดีไซน์ Neubrutalist UI**:
   - ใช้ขอบสีดำสนิทที่มีความหนาชัดเจน (`border-[2px] border-black` หรือ `border-[4px] border-black`)
   - ใช้เงาพิกเซลสีดำเรียบตรงทึบ ไม่มีเอฟเฟกต์ฟุ้งหรือเบลอ (`shadow-[4px_4px_0px_#1E1E1E]`)
   - ใช้เฉดสีพาสเทลสะท้อนแสงสว่างสดใสเป็นพื้นหลังในการ์ดและปุ่มต่างๆ
2. **การป้องกัน XSS**: ทุกครั้งที่จะนำค่าของตัวแปรหรือข้อมูลจากภายนอกมาแสดงใน HTML โดยใช้ `innerHTML` จะต้องห่อตัวแปรด้วยฟังก์ชัน `escapeHTML()` เพื่อกรองโค้ดอันตรายเสมอ
3. **ระบบความปลอดภัยของ Supabase RLS & JWT**: ในระบบฐานข้อมูลมีการตั้งค่าความปลอดภัยระดับแถว (Row-Level Security หรือ RLS) ขอแนะนำให้ดึงข้อมูลผ่านนโยบายความปลอดภัย RLS บนระบบ Supabase เสมอเพื่อความเสถียรและปลอดภัยในการแสดงผลข้อมูลตามสิทธิ์
4. **คำสั่งในการพัฒนาโปรเจกต์ (Vite)**:
   - คำสั่งติดตั้งไลบรารี: `npm install`
   - คำสั่งรันระบบเพื่อการพัฒนาในเครื่อง: `npm run dev`
   - คำสั่งบิลด์ไฟล์เพื่อพร้อมสำหรับโปรดักชัน: `npm run build`
   - คำสั่งจำลองพรีวิวผลงานหลังการบิลด์: `npm run preview`
