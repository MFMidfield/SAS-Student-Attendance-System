# รายการหน้าเพจที่สามารถทำเป็น Shared Module ได้ (เหมือนหน้า Activity)

จากการวิเคราะห์โครงสร้างโฟลเดอร์ใน `src/pages/` ของแต่ละระบบ (Admin, Teacher, Leader, Student) พบว่ามีอีกหลายหน้าที่มีโครงสร้างการทำงานและหน้าตา (UI) คล้ายคลึงกัน ซึ่งสามารถนำมาทำเป็น **Shared Component / Reusable Module** แบบเดียวกับหน้า Activity ได้ เพื่อลดความซ้ำซ้อนของโค้ด (Code Duplication) ให้เหลือน้อยที่สุด

นี่คือรายการหน้าเพจที่แนะนำให้ยุบรวม พร้อมคำอธิบายโดยละเอียดครับ:

---

## 1. หน้าตั้งค่าบัญชี (Setting Page)
**ไฟล์ปัจจุบัน:** 
- `src/pages/admin/setting/`
- `src/pages/teacher/setting/`
- `src/pages/leader/setting/`
- `src/pages/student/setting/`

**เหตุผลที่ควรยุบรวม:**
หน้า Setting ของทุก Role มักจะมีฟังก์ชันพื้นฐานที่เหมือนกัน 100% เช่น:
- การเปลี่ยนรูปโปรไฟล์ (Avatar)
- การดูข้อมูลส่วนตัวเบื้องต้น
- การเปลี่ยนรหัสผ่าน
- ปุ่มออกจากระบบ (Logout)

**แนวทางการ Refactor:**
- **สร้างไฟล์กลาง:** `src/pages/shared/setting/setting.html` และ `setting.js`
- **JS Logic:** สร้างฟังก์ชัน `initSetting(role, userAvatarUrl)`
- หน้าตาสามารถใช้เทมเพลตเดียวกันได้เลย ไม่ต้องซ่อน/แสดงปุ่มอะไรซับซ้อน ยกเว้นปุ่ม Back ที่จะให้กลับไปที่ Dashboard ของ Role นั้นๆ (`#${role}-dashboard`)

---

## 2. หน้าตารางเวลา/ตารางเรียน (Schedule Page)
**ไฟล์ปัจจุบัน:**
- `src/pages/admin/schedule/`
- `src/pages/teacher/schedule/`
- `src/pages/leader/schedule/`
- `src/pages/student/schedule/`

**เหตุผลที่ควรยุบรวม:**
หน้าตารางเรียนหรือตารางสอนมีโครงสร้างปฏิทินหรือการแสดงผลแบบตารางที่เหมือนกัน แต่จะต่างกันที่ "สิทธิ์ในการแก้ไข" และ "ข้อมูลที่ถูกดึงมาแสดง":
- **Admin / Teacher:** อาจจะมีปุ่มสำหรับแก้ไขตารางสอน, เพิ่มวิชาเรียน, ยกเลิกคลาส
- **Leader / Student:** มีสิทธิ์แค่ดู (Read-Only) ตารางเรียนของตัวเองหรือของห้อง

**แนวทางการ Refactor:**
- **สร้างไฟล์กลาง:** `src/pages/shared/schedule/schedule.html` และ `schedule.js`
- **JS Logic:** สร้างฟังก์ชัน `initSchedule(role, userId)`
- เพิ่มเงื่อนไข `const hasEditAccess = (role === 'admin' || role === 'teacher');` เพื่อซ่อน/แสดงปุ่มเพิ่มหรือแก้ไขตารางเรียน
- การ Query ข้อมูล: หากเป็น Student ให้ดึงเฉพาะตารางของห้องตัวเอง หากเป็น Teacher ให้ดึงตารางที่ตัวเองสอน

## 3. หน้าจัดการผู้ใช้งาน (User Management Page)
**ไฟล์ปัจจุบัน:**
- `src/pages/admin/user/`
- `src/pages/teacher/user/`
- `src/pages/leader/user/`

**เหตุผลที่ควรยุบรวม:**
หน้านี้ใช้สำหรับแสดงรายชื่อผู้ใช้ (ครู, นักเรียน) ซึ่งหน้าตาตารางรายชื่อและช่องค้นหา (Search) จะเหมือนกันเป๊ะ สิ่งที่ต่างคือสิทธิ์:
- **Admin:** เห็นทุกคน และสามารถ เพิ่ม/ลบ/แก้ไข ข้อมูลผู้ใช้ทุกคนได้ (Full CRUD)
- **Teacher:** อาจจะเห็นเฉพาะนักเรียนในที่ปรึกษา และแก้ไขข้อมูลได้บางส่วน (เช่น เบอร์ติดต่อผู้ปกครอง)
- **Leader:** เห็นรายชื่อเพื่อนในห้อง แต่ไม่มีสิทธิ์แก้ไขใดๆ (Read-Only)

**แนวทางการ Refactor:**
- **สร้างไฟล์กลาง:** `src/pages/shared/user/user_management.html` และ `user_management.js`
- **JS Logic:** ตรวจสอบ `role` เพื่อควบคุม:
  1. ควบคุมการแสดงปุ่ม "Add User" (โชว์เฉพาะ Admin/Teacher)
  2. ควบคุมปุ่ม "Edit/Delete" หลังรายชื่อแต่ละคน (โชว์เฉพาะผู้ที่มีสิทธิ์)
  3. API Query: ใช้ RLS (Row Level Security) หรือกำหนดเงื่อนไขในการดึงลิสต์รายชื่อจาก Supabase ให้ตรงกับ Role

## 4. หน้าอนุมัติการมาเรียน (Attendance Approve Page)
**ไฟล์ปัจจุบัน:**
- `src/pages/admin/attendance_approve/`
- `src/pages/teacher/attendance_approve/`
- `src/pages/leader/attendance_approve/`

**เหตุผลที่ควรยุบรวม:**
หน้านี้เป็นหน้าสำหรับตรวจสอบคำขอลา (ป่วย, ลากิจ) หรือการแก้ไขประวัติการเข้าเรียน โครงสร้าง UI จะเป็นการ์ดแสดงคำขอและมีปุ่ม "Approve" (อนุมัติ) และ "Reject" (ปฏิเสธ)
ซึ่งทั้ง Admin, Teacher และ Leader ทำงานผ่าน UI เดียวกันทั้งหมด

**แนวทางการ Refactor:**
- **สร้างไฟล์กลาง:** `src/pages/shared/attendance_approve/approve.html` และ `approve.js`
- **JS Logic:** ฟังก์ชัน `initApprove(role)`
- หน้า UI แทบไม่ต้องซ่อนปุ่มใดๆ เพราะทั้ง 3 Role มีสิทธิ์กดยืนยัน/ปฏิเสธ แต่จะต่างกันที่ระดับการอนุมัติ (เช่น Leader อนุมัติเบื้องต้น -> Teacher อนุมัติขั้นสุดท้าย) 
- ใช้ JS เช็ก `role` เพื่อเปลี่ยน Status เวลาอัปเดตข้อมูลลงฐานข้อมูล (เช่น หาก Leader กด จะเปลี่ยนสถานะเป็น `leader_approved` แต่ถ้า Teacher กด จะเปลี่ยนเป็น `approved_final` เป็นต้น)

## 5. หน้าบันทึกการมาเรียน (Attendance Page - สำหรับ Student และ Leader)
**ไฟล์ปัจจุบัน:**
- `src/pages/student/attendance/`
- `src/pages/leader/attendance/`

**เหตุผลที่ควรยุบรวม:**
หน้าจดบันทึกการเช็กชื่อ/การลาของ Student และ Leader มีขั้นตอนและตรรกะเบื้องหลังเหมือนกัน 100% (เช่น การเช็กข้อมูลผู้ใช้งาน, การหาตารางเรียนปัจจุบัน, การเช็กขีดจำกัดการลาต่อวัน, และการกดยืนยันข้อมูลลงตาราง `attendance_logs`) เนื่องจาก Leader (หัวหน้าห้อง) ก็เป็นหนึ่งในนักเรียนเช่นเดียวกัน

**แนวทางการ Refactor:**
- **สร้างไฟล์กลาง:** `src/pages/shared/attendance/attendance.html` และ `attendance.js`
- **JS Logic:** สร้างฟังก์ชัน `initAttendance(role, imageLogo, imageBander)`
- **การแยกบทบาทเมื่อส่งข้อมูล (Submit):** 
  - **การนำทาง (Navigation):** ใช้ตัวแปร `role` เพื่อกำหนดให้ปุ่ม Back ลิงก์กลับไปยังหน้า Dashboard ของ Role ตนเองโดยอัตโนมัติ (`window.location.hash = '#' + role + '-dashboard'`)
  - **การส่งข้อมูลและบันทึก:** ไม่จำเป็นต้องมีเงื่อนไขแยกสิทธิ์ใดๆ ฝั่งหน้าบ้าน (Client) ตอนยิงไปบันทึก เพราะ Supabase จะเก็บสิทธิ์และรหัสผู้ใช้ผ่าน Token ล็อกอินที่ปลอดภัยอยู่แล้ว ค่า `student_id` จะเชื่อมต่อไปที่ผู้ใช้นั้นๆ โดยตรง

## 6. 🛡️ อธิบายเรื่องความปลอดภัยและการป้องกันการโดนแฮ็ก (Security Design)

การยุบรวมไฟล์ (Shared Module) จะทำให้โค้ดที่รันในทุก Role คือไฟล์เดียวกัน ดังนั้นหากมีคนที่ไม่หวังดีพยายามจะ "แฮ็ก" ด้วยการลบ Class ที่ซ่อนปุ่มออก (ลบ `hidden` ใน DevTools) เพื่อแก้ไขข้อมูลคนอื่น เราจะต้องมีการป้องกันอย่างรัดกุมแบบ **2 ชั้น (2-Layer Security)** ดังนี้:

### ชั้นที่ 1: การป้องกันฝั่งเบราว์เซอร์ (Frontend Logic)
บนเว็บไซต์ (Client-side) เราจะไม่เชื่อใจผู้ใช้ (Never trust the client)
- **ไม่ผูก Event Listener ลอยๆ:** ถึงแม้แฮกเกอร์จะเสกปุ่ม "เพิ่มข้อมูล" ขึ้นมาในหน้าเว็บได้ แต่ถ้าเราเขียนโค้ดผูกการคลิกปุ่มไว้ในเงื่อนไขสิทธิ์ `if (hasWriteAccess) { btnAdd.addEventListener(...) }` แฮกเกอร์กดปุ่มไปก็จะไม่เกิดอะไรขึ้นเลย เพราะเบื้องหลังปุ่มนั้นไม่มีฟังก์ชันมารองรับ
- **ไม่มีการเรียก API ถือว่าปลอดภัย:** เมื่อกดปุ่มไม่ได้ ก็จะไม่มีการดึงหน้าต่าง Modal ขึ้นมา และไม่มีคำสั่งเรียกไปหา Supabase

### ชั้นที่ 2: การป้องกันที่ฐานข้อมูลด่านสุดท้าย (Backend & Supabase RLS) 🔒 (สำคัญที่สุด)
นี่คือจุดที่ปลอดภัยที่สุดของการทำแอปด้วย Supabase ต่อให้แฮกเกอร์ข้ามการกดปุ่มหน้าเว็บไป แล้วเขียนโค้ดยิง API (Fetch request) ใน Console ของเบราว์เซอร์เพื่อพยายามลบ/แก้ไขข้อมูล ฐานข้อมูลจะป้องกันไว้ด้วย **Row Level Security (RLS)**
* **RLS ทำงานอย่างไร?**
  ทุกครั้งที่มี Request เข้ามา Supabase จะแกะดู Token การล็อกอิน ว่า **"คนนี้คือใคร และมี Role อะไร"** (Token นี้เข้ารหัสปลอมแปลงไม่ได้)
* **ตัวอย่างการตั้ง Policy:**
  - `SELECT (อ่านข้อมูล):` ทุกคนที่มี Token (ล็อกอินแล้ว) เข้าถึงได้
  - `INSERT / UPDATE / DELETE (แก้ไขข้อมูล):` อนุญาตเฉพาะผู้ใช้ที่มี role เป็น `admin` หรือ `teacher` เท่านั้น
* **ผลลัพธ์การแฮ็ก:** ถ้าคนที่เป็น `leader` ยิง API ขอแก้ข้อมูลเข้า Supabase ตัวฐานข้อมูลจะเช็ก Token แล้วพบว่าสิทธิ์ไม่ถึง จากนั้นจะ **ดีด Request ทิ้งทันที (Permission Denied 403)** ข้อมูลจึงปลอดภัย 100%

### สรุปความปลอดภัย:
การรัน Module เดียวกัน ไม่ได้แปลว่าเว็บเราจะโดนแฮ็กได้ง่ายขึ้น ตราบใดที่เรา **ซ่อนปุ่มที่หน้าบ้านให้เนียน** และมี **RLS ป้องกันหลังบ้าน** จะทำให้เว็บของเราเขียนโค้ดได้สั้นลงและมีความปลอดภัยสูงเทียบเท่าเว็บระดับโลกครับ
